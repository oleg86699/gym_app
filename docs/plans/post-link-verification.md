# Валидация ссылки в опубликованном посте

Цель: после успешного постинга подтверждать, что НАШ бэклинк реально виден на
живой странице поста (а не только «WP вернул post_id»). Резолвить реальный
permalink (follow redirects) и сохранять его.

## Решения (подтверждены пользователем)
- **2 режима** в форме создания (post-типы): `mark` (Отметка) / `auto` (Автовалидация).
  Проверка делается ВСЕГДА (нет режима «совсем без проверки»).
- **mark**: 1 GET → отметка ✓/✗ + резолвленный URL. Пост считается done в любом случае.
- **auto**: GET; если ссылки нет — перечитать тот же пост ещё пару раз (кэш/задержка),
  потом перепостить на другой сайт (hopping), и так до подтверждения / исчерпания сайтов.
  Пост засчитывается только после подтверждения.
- **Критерий «ссылка есть»**: достаточно домена ссылки (любой href на target_domain).

## Хранение (как есть сейчас)
- `text_items.post_id` (BigInt) и `posted_url` (синтетический `{host}/?p={id}`) — хранятся.
- `verified_at` + `verified_urls` (JSONB) уже есть (link-типы) → переиспользуем.
- Хелперы `_anon_get` (CF-fallback) и `_html_has_link` в wp_admin_client — переиспользуем/копируем.

## Примечания
- auto-режим ограничен общим per-item бюджетом попыток (~10 сайтов) — если ни на одном
  сайте ссылка не подтвердилась, айтем остаётся pending → need_more_admins (как обычно).
  В худшем случае это до ~10 «осиротевших» постов (сайт принял пост, но вырезал ссылку).
- Резолвленный permalink сохраняется в `posted_url` (заменяет синтетический `?p=id`).
- spin_fanout-раны получают `mark` по умолчанию (модельный default), отдельного селектора пока нет.
- Живая проверка полного цикла (реальный постинг + GET) — на реальном прогоне.
