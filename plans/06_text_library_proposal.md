# Этап 6 (proposal). Унифицированная библиотека текстов + ссылки/анкоры + поиск + аналитика

> Статус: **черновик на согласование**. Код не пишем, пока не утвердим.
> Документ собирает всё, что обсудили: единое хранилище текстов из разных
> источников, генератор (позже), переиспользование, поиск по анкору, разбор
> ссылок из текстов и аналитику по доменам.

## Цель

Сделать так, чтобы **все тексты из любых источников попадали в одну
унифицированную таблицу** и использовались оттуда для **всех вариаций
простановки** (обычный пост, сквозная, с главной, и будущие). Поверх этого:

- **генератор** (позже) пишет сгенерированные тексты в ту же таблицу;
- **режим reuse** — переиспользовать старые тексты из этой же базы, когда
  качество не критично;
- **поиск текста по анкору** (нечёткий) — под будущие задачи, где даётся
  только `link + anchor`;
- **аналитика** — сколько ссылок/текстов размещено на какой домен и на какой
  донор-сайт/страницу.

Опираемся на проверенную боевую систему пользователя
(`gym_gen_content_casino`): тексты там уже хранятся **в Postgres**
(`content_new_en_test_index`: `clear_content` + `clear_content_tsv` + `spin_content`),
поиск — **полнотекст Postgres** (`phraseto_tsquery` / `ts_headline`), вставка
ссылки — `replace_links()` с типами `anchor | non anchor | non anchor link`,
уникализация — spintax. Эту модель и берём за основу.

---

## 1. Текущее состояние (как сейчас в `app`)

- Тело текста → объект в **MinIO** (`text-items` bucket), ключ `storage_key`.
- Метаданные → `text_items` (per-run задача): `title`, `content_hash`,
  `byte_size`, `status`, `site_id` (донор-сайт), `post_id`, **`link_url`**,
  **`link_anchor`**, `placed_via`, `placement_ref`, `verified_*`, `not_before`
  (drip), `posted_at`.
- `posting_runs` — прогон; `projects` — проект (пока без списка доменов).
- Язык **сайта** уже определяется при валидации → `wp_sites.language`.

Вывод: per-task `link/anchor` **уже есть** в `text_items`. Не хватает: единой
библиотеки текстов, языка текста, доменов проекта, разбора ссылок и аналитики.

---

## 2. Зафиксированные решения (decisions log)

1. **Тела текстов под библиотеку — в Postgres, не в MinIO.** Так делает
   боевая система; объём мелкий (100k × ~12KB ≈ 1.2 ГБ); reuse/поиск/
   консистентность сильнее в одном транзакционном хранилище. MinIO под тексты
   выводим в legacy (остаётся для архивов/картинок).
2. **Поиск — БЕЗ ЛЛМ.** Основа — полнотекст Postgres (`tsvector`) +
   `pg_trgm` (нечёткий анкор) + `pgvector` (смысловой, hybrid) — всё в Фазе B.
   ЛЛМ в базе ничего не «ищет»; embedding лишь конвертит «текст → числа»,
   поиск — обычный SQL внутри Postgres.
3. **Язык текста — обязателен, но пока ТОЛЬКО хранить** (для будущего поиска).
   `langdetect` на заливке txt; в генераторе — селектор языка.
4. **Убрали** `link_type` (старая логика) и `count` как множитель — каждый
   текст = **отдельная строка** (`count=N` раскрываем в N задач).
5. **Не создавать `undefined`-строки** для текстов без распознанного домена.
   Отсутствие связи = «бэклинк не найден» — это уже значимо. Вместо пустышки —
   статус `needs_review` + дозаполнение (см. §5.3).

## 3. Целевая модель данных

```text
texts                          -- единая библиотека контента
  id
  body            TEXT         -- = clear_content
  body_tsv        tsvector     -- GIN-индекс, FTS-поиск по ключу/теме
  spin_body       TEXT NULL    -- spintax-версия (уникализация при reuse)
  title           TEXT
  lang            varchar(10)  -- ← НОВОЕ (langdetect / селектор генератора)
  source          enum(human | generated | reused)
  gen_model       varchar NULL -- gpt_model / prompt_name (как в CSV генератора)
  content_hash    char(64)     -- дедуп
  times_used      int default 0
  created_at, last_used_at, archived_at   -- lifecycle (тексты не вечны)

text_links                     -- ссылки/анкоры, извлечённые/заданные (1 текст → N)
  id
  text_id         → texts.id   (или text_item_id для runs-flow, см. §6)
  link            TEXT         -- целевой URL
  anchor          TEXT
  target_domain   varchar      -- нормализованный домен из link (для аналитики)
  is_project_domain bool       -- входит ли target_domain в project_domains
  created_at

project_domains                -- домены, которые продвигает проект (money/target)
  id, project_id → projects.id, domain, created_at
  UNIQUE(project_id, domain)

-- text_items (УЖЕ ЕСТЬ) — задача в runs. Добавляем:
  + target_domain   varchar NULL   -- нормализованный домен из link_url (аналитика)
  + status 'needs_review'          -- задача требует дозаполнения (см. §5.3)
  + link_candidates jsonb NULL     -- извлечённые ссылки-кандидаты для UI выбора
  + text_id → texts.id NULL        -- связь с библиотекой (Фаза B)
```

Индексы: `GIN(body_tsv)`; `GIN(... gin_trgm_ops)` на `anchor`/`title`;
`(project_id, domain)` уникальный; `text_links(target_domain)`,
`text_links(text_id)`.

## 4. Источники добавления (всё пишет в одни таблицы)

| Источник | Что делаем |
|---|---|
| **txt-файлы (как сейчас)** | парсим HTML, извлекаем все `<a href>`+анкор; определяем целевой домен по `project_domains` (см. §5); пишем `texts`(+`lang`) и `text_links`; неоднозначные → `needs_review` |
| **csv/xlsx: `link, anchor, text`** | напрямую: `texts.body = text` (+detect `lang`); `text_links(link, anchor, target_domain=домен из link, is_project_domain)` |
| **csv-кампания: `links, anchor, counts, type`** (текущий формат SEO) | строки = задачи: `counts` раскрываем в N задач `anchor → link`; текст под них **генерируется** или берётся из библиотеки (reuse) |

Домен везде вычисляем из `link` → `target_domain` → база для аналитики.

---

## 5. Разбор ссылок из текста + disambiguation + needs_review

### 5.1 Предложение по проекту
К проекту **обязательно вводится список доменов** (`project_domains`) — это
**целевые (money) домены**, которые проект продвигает (на них ведут анкоры).
По ним из txt вытаскиваем именно «наши» бэклинки и строим аналитику.

### 5.2 Правило определения (ключ — число РАЗНЫХ «наших» доменов в тексте)
Извлекаем все `<a href>`; «наши» = те, чей домен ∈ `project_domains`.

| Ситуация | Действие |
|---|---|
| **ровно 1 «наш» домен** (ссылок на него хоть несколько; + любые ссылки на источники) | ✅ авто: домен однозначен → `target_domain` + `link_url`/`link_anchor`; источники сохраняем `is_project_domain=false`, в атрибуции игнор |
| **≥2 разных «наших» домена** | ✅ **берём первый** «наш» как целевой (дефолт), задача готова; можно переопределить вручную в UI |
| **0 «наших» доменов** (ссылки есть, но не наши) | ⚠️ `needs_review`, причина «домен не в проекте» → показать ссылки, кнопка «добавить домен в проект» / выбрать вручную |
| **ссылок нет вообще** | ⚠️ `needs_review`, причина «нет ссылки» → дозаполнение (см. §5.3) |

Ссылки на источники сами по себе **не ломают** определение (они не-проектные).
*(решение по ≥2: берём первый — подтверждено.)*

### 5.3 Состояние «нужны данные»
- Статус задачи **`needs_review`** + причина + `link_candidates` (jsonb) для UI.
- **Гейт постинга**: `_pick_pending_batch` и подбор сайтов **пропускают**
  `needs_review` — такие задачи в постинг **не идут**, пока не заполнят.
- **UI «дозаполнить»** (вкладка/фильтр «Needs data» на странице run-а). Юзер
  резолвит задачу любым из путей:
  1. выбрать целевую ссылку из кандидатов (если они есть);
  2. **вписать `link`/`anchor` в поля задачи** вручную;
  3. **отредактировать сам текст** — добавить/поправить `<a>` в теле.
  **На сохранении** (поля или текст) — пере-извлекаем ссылки и проверяем: если
  появился валидный «наш» бэклинк → статус → `pending` → постится. Если нет —
  остаётся `needs_review`.
- **Авто-резолв**: при добавлении домена в проект — пере-сканировать
  `needs_review`; совпавшие авто-резолвятся (массовое закрытие «домена не было»).

---

## 6. Связь библиотеки и runs

- `texts` — источник (тело + язык + tsv + spin).
- `text_items` (задача в runs) — ссылается на `texts.id` (Фаза B) и держит
  per-task `link_url`/`link_anchor`/`target_domain`/`site_id`/`post_id`.
  Один и тот же текст библиотеки можно переиспользовать в разных задачах с
  разными анкорами (поэтому link/anchor живут на задаче, а не только на тексте).
- `count=N` из кампании → N строк `text_items` (drip-feed `not_before` уже
  умеет разносить их по дням — см. этап drip).

---

## 7. Язык

- `wp_sites.language` — **уже есть** (определяется на валидации).
- Добавляем `texts.lang` (langdetect на заливке; генератор — из селектора).
- **«Гейт по языку»** = при подборе донор-сайта брать только сайты с
  `site.language == text.lang` (англ. текст → только англ. доноры). Половина
  готова (язык сайта есть).
- **Решение**: в Фазе A — **только хранить `lang`**. Сам подбор по языку делаем
  позже **как параметр задачи** (per-run выбор языка доноров) — подтверждено
  пользователем.

## 8. Поиск (под reuse и будущие задачи)

1. **FTS**: `body_tsv @@ phraseto_tsquery(анкор/ключ)` — основной, проверенный
   (как `content_new_en_test_index`). `ts_headline` — для заголовка под вставку.
2. **`pg_trgm`** поверх — «анкор написан по-разному» (нечёткое совпадение).
3. **`pgvector`** (смысловой/embeddings) — **закладываем в Фазе B** вместе с
   поиском (hybrid: FTS+trgm+vector с самого старта). Ловит случаи, где слова
   разные, а смысл тот же (FTS по словам их пропускает). Решение — включаем,
   т.к. объём быстро растёт. Запрос лёгкий (HNSW); цена — embedding-API на
   вставке (~$1/100k текстов) + хранение вектора.
4. Уникализация при reuse — `spin_body` (spintax), перенос `replace_links()`.

---

## 9. Аналитика

Связка `texts ↔ text_links(target_domain) ↔ text_items(site_id, post_id,
verified_urls)` даёт оба разреза:

- **по target-домену**: сколько бэклинков/текстов размещено на `nawal.mx` и т.п.;
- **по донор-сайту/странице**: сколько наших ссылок ушло на каждый сайт пула.

## 10. Фазирование

### Фаза A — можно сейчас, ценность сразу, без генератора, без трогания MinIO
- `project_domains` (таблица + UI ведения списка доменов проекта);
- извлечение `<a href>`/анкор из txt при заливке + **disambiguation** (§5);
- `text_items`: `target_domain`, статус `needs_review`, `link_candidates`;
- **needs_review-гейт** в постинге + **UI дозаполнения** + авто-резолв;
- `texts.lang` пока как поле на `text_items` (язык текста) — детект на заливке;
- аналитика по `target_domain` на текущих runs.

> Фаза A самостоятельна и низкориска: даёт аналитику по доменам и разбор
> ссылок/needs_review на текущем пайплайне, не мигрируя хранилище.

### Фаза B — вместе с генератором (позже)
- унифицированная `texts` (тела → в БД, `body_tsv` + `spin_body` + `source` +
  `content_hash` + lifecycle); миграция тел из MinIO;
- FTS + `pg_trgm` поиск, режим reuse, `pgvector` (опц.);
- входы `csv-direct (link,anchor,text)` и `кампания (links,anchor,counts,type)`;
- генератор пишет в `texts`; связь `text_items.text_id → texts.id`.

### Совместимость
- Всё аддитивно. `spread_days=0`/без `project_domains`/без `needs_review` →
  поведение как сейчас. Миграция тел из MinIO — отдельный аккуратный шаг Фазы B
  (двойное чтение `text_id ?? storage_key` на переходный период).

## 11. Решения (согласовано)

1. ✅ **`project_domains` = целевые (money) домены проекта** (на которые ведут
   анкоры). Доноры — это сайты из `wp-sites`, откуда мы постим; это разное.
2. ✅ **Текст без ссылок** → `needs_review` + дозаполнение: вписать `link/anchor`
   в поля задачи **или** добавить ссылку в текст; на сохранении пере-проверяем,
   ок → постим (§5.3).
3. ✅ **≥2 разных «наших» доменов** → **берём первый** (дефолт), с возможностью
   переопределить вручную. Не блокируем.
4. ✅ **Язык**: в Фазе A только **хранить `lang`**. Подбор доноров по языку —
   позже, **как параметр задачи** (per-run выбор языка доноров).
5. ✅ **`pgvector`** — **закладываем в Фазе B** (hybrid FTS+trgm+vector сразу),
   т.к. объём быстро растёт; нагрузка на запросах низкая.

## Приложение. Форматы входных файлов (текущие у SEO)

- **Кампания** (скрин): `links, anchor, counts, type` —
  `links=https://nawal.mx/`, `anchor` (варианты), `counts` (сколько простановок),
  `type=anchor`.
- **txt** с готовым текстом, внутри `<a target="_blank" rel="noopener noreferrer
  nofollow" href="https://target/">анкор</a>` — извлекаем `link+anchor`.
- **csv/xlsx direct** (предложено): `link, anchor, text`.
